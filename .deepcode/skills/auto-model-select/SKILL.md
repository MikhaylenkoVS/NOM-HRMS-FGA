# Auto Model Select

Автоматически подбирает оптимальную модель и глубину reasoning под характер запроса, чтобы не тратить токены на тривиальные задачи и не экономить на сложных.

## Правила выбора

### 1. Не переопределять явный выбор

- Если пользователь вручную вызвал `/model` — **не менять** модель до конца сессии (или явной отмены).
- Если в текущей сессии уже была смена модели через `/model` — сохранять выбор пользователя.

### 2. Режим при старте сессии

**По умолчанию сессия стартует на Flash** (`deepseek-v4-flash`, thinking off). Модель повышается (Pro High / Pro Max) только когда задача выходит за пределы возможностей Flash — см. правила ниже.

### 3. Алгоритм автоматического выбора

Перед каждой задачей оцени запрос по трём критериям и выбери модель + reasoning:

| Уровень | Модель | Reasoning | Когда |
|---------|--------|-----------|-------|
| 🟢 **Flash** | `deepseek-v4-flash` | — (thinking off) | Малые/тривиальные задачи: ответы на вопросы без кода, поиск/навигация (`найди`, `покажи`, `где`), правки ≤10 строк в некритичных файлах, добавление import/аннотаций/docstring, переименования, создание файлов по шаблону, простые shell-команды. |
| 🟡 **Pro High** | `deepseek-v4-pro` | `high` | Стандартные задачи: правки >10 строк кода, написание новых функций/классов, создание/обновление тестов, исправление багов в бизнес-логике, рефакторинг с умеренным объёмом изменений, правки в критичных зонах (security, ядро, data integrity). |
| 🔴 **Pro Max** | `deepseek-v4-pro` | `max` | Сложный анализ: code review, архитектурные решения, отладка критических багов, планирование (Plan Mode), рефакторинг с побочными эффектами, многомодульные изменения, запрос >800 символов с глубоким контекстом. |

### 4. Ключевые слова-триггеры для Pro Max

Если в запросе есть **любое** из:
- `code review`, `review`, `ревью`, `анализ`, `analyse`, `analyze`
- `архитектур`, `architecture`, `design`
- `debug`, `отладка`, `critical`, `критический`
- `plan`, `план`, `milestone`
- `безопасность`, `security`
- Запрос явно просит глубокого анализа или сравнения альтернатив

→ **Pro Max** (`deepseek-v4-pro` + `reasoningEffort: "max"`)

### 5. Когда выбирать Flash

**Принцип**: если задачу способен сделать джун без присмотра → Flash.

Конкретные индикаторы (достаточно **любого**):

- **Нет кода**: ответ на вопрос, объяснение, справка — и никакой код не пишется.
- **Поиск/навигация**: `найди`, `покажи`, `где`, `какой файл`, `find`, `grep`, `ls`.
- **Малая правка** (≤10 строк изменений) **в некритичном файле**:
  - Добавление/удаление import
  - Добавление docstring, комментария, type annotation
  - Переименование переменной/функции/класса
  - Исправление опечатки в строке/имени
  - Правка одного-двух условий или значений
- **Создание простого файла** по шаблону (навык, конфиг, boilerplate).
- **Shell-команды**: `ls`, `find`, `git status`, `git diff`, `pip list` и аналоги.
- Запрос ≤300 символов и не содержит слов-триггеров Pro Max/Pro High.

**Граница с Pro High**: если правка требует >10 строк нового кода, затрагивает критичную зону (ядро приложения, security, data integrity, конфигурация прода) или пользователь явно просит «напиши функцию/класс» → Pro High.

### 6. Когда выбирать Pro High

**Pro High — когда Flash не справляется.** Выбирается, если:

- Правка >10 строк кода или создание нового кода.
- Правка в критичной зоне: security, auth, ядро, data integrity, prod-конфигурация.
- Запрос содержит: `напиши`, `создай функцию`, `добавь функциональность`, `исправь баг`, `рефакторинг`.
- Написание или обновление тестов.
- Запрос 300–800 символов с контекстом.
- Не подходит ни под Flash, ни под Pro Max.

`deepseek-v4-pro` + `reasoningEffort: "high"`

## Протокол смены модели

При срабатывании правила:

1. Вызови `/model` и выбери нужную комбинацию.
2. В начале ответа кратко укажи: `[→ pro/max]`, `[→ pro/high]` или `[→ flash]`.
3. Если модель уже та же, что выбрана — не переключай.
